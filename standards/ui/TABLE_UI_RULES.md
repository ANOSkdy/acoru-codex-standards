# TABLE_UI_RULES

## 目的
一覧画面で毎回別流儀にならないよう、テーブルの情報設計・表示・操作ルールを固定します。

## テーブルを使う条件
次の条件を満たすとき、card 一覧より table を優先します。

- 行の比較が重要
- 列ごとの意味が安定している
- ソート / フィルタ / bulk 操作が必要
- 行数がある程度多い

## 基本構成
1. ページタイトル
2. 補助説明
3. primary / secondary action
4. フィルタ / 検索
5. table
6. pagination または load more
7. empty / error / loading state

## 列設計
### 列順
基本順:
1. 主識別子
2. 補助識別子
3. status
4. 所有者 / 関係先
5. 主要日時
6. 補助日時
7. 行アクション

### 列幅
- 主識別子は広め
- status, 数値, 日付は狭め
- 長文は省略表示を前提にする
- 同一table内で意味なく可変幅を増やさない

### 揃え
- テキストは左揃え
- 数値は右揃え
- 日付は左または中央で統一
- status badge は中央寄せにしない。読み順を優先する

## 表示ルール
- null は `—` で表す
- 空文字は原則表示しない
- 日時フォーマットは PJ で固定する
- 金額や件数は桁区切りを入れる
- 長いIDは必要なら短縮表示＋詳細導線
- 同じ値の意味が画面で変わらないようにする

## status 表示
- badge / chip の表現は共通
- 色だけでなく文言を表示する
- 同じ status 値は同じラベルと色で表す

## 行アクション
- 行全体クリックは詳細遷移が主目的のときだけ
- 破壊的操作は row の即時 action に置かない
- その他メニューで隠す場合も、主要導線は分かりやすくする

## ソートとフィルタ
- デフォルト並び順を明示する
- ソート可能列は本当に意味があるものだけ
- フィルタ値は docs の enum / status と一致させる
- フィルタUIで自由文字列を enum の代わりに使わない

## loading / empty / error
### loading
- skeleton を優先
- 行数は想定件数感に近づける
- spinner だけで高さが大きく変わらないようにする

### empty
- 「何もない」だけで終わらせない
- 次に取れる行動を1つ出す
- フィルタ適用中の empty と初期 empty を分ける

### error
- 一覧全体エラーと行アクションエラーを分ける
- 再試行導線を置く
- 内部エラー詳細を露出しない

## pagination
- 件数が多い場合は pagination を基本とする
- デフォルト件数は PJ で固定する
- URL と同期するなら query param 名も固定する
- 無限スクロールは明確な理由がある場合のみ

## responsive
- モバイルで比較が重要なら横スクロールを許容
- モバイルで比較不要なら card へ変換してよい
- どちらにするかは table ごとに `TABLE_SPECS.md` に明記する

## 監査観点
- 列順が一貫しているか
- null 表記が揃っているか
- status 表現が揃っているか
- ソート初期値が妥当か
- 行アクションが危険すぎないか
